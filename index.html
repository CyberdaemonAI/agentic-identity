Securing Agentic Identity

What commercial platforms give you, what open standards add, and what none of them address yet.

01 / The Problem Space 200

Traditional IAM assumes the principal is a human. Agentic AI breaks that assumption at every layer.

The tools exist. The frameworks exist. The problem is that none of them were designed for a principal that spawns sub-principals, holds persistent memory across sessions, calls external tools autonomously, and can be prompted into acting outside its declared purpose.

Credential Sprawl

Each agent, service connection, and tool integration creates a new identity. Enterprise deployments generate hundreds of non-human credentials with no lifecycle management. When the agent is decommissioned, the credentials often are not.

Absent Scope Binding

Platforms issue credentials at service level, not session level. A customer service agent runs with the same credential whether answering an FAQ or attempting a database write. The session intent is invisible to the authorization system.

Delegation Chain Collapse

When Agent A delegates to Agent B, which delegates to Agent C: which principal authorized C's action? Existing IAM models assume one authorization event per session, not a chain. Multi-agent systems blow past this assumption constantly.

Persistent State Authorization

Agent memory persists across sessions. No platform has an authorization model for reads and writes to that memory. What governs which agent can write what to a shared memory store? Currently: nothing formal.

Behavioral Drift

An agent's declared intent ("I will summarize emails") may diverge from observed behavior at runtime. Static credentials cannot detect or block this drift. By the time a log surfaces an anomaly, the action has already executed.

"The question is not just who this agent is. It is whether this agent is doing what it said it would do, within the scope it was granted, right now."

02 / Commercial Platform Coverage 300

What the major platforms actually give you

Each platform below has a real identity model. None of them solve the session-scope binding problem or behavioral authorization. That is not a criticism, it is an accurate description of the current state.

Microsoft Copilot / AI Foundry
Microsoft
Identity Model
Microsoft Entra ID. Agents run as managed identities or service principals. Delegated permissions allow agents to act on behalf of users within defined OAuth scopes.
Credential Type
Managed Identity tokens (Azure AD). Short-lived (~1hr), auto-rotated by the platform.
Scope Binding
OAuth 2.0 scopes at service level. Copilot Studio connectors use separate connection credentials per connector type.
Delegation
Delegated permissions chain through Entra. Multi-agent: sequential handoff via Teams channel or direct API with service principal.
Audit
Microsoft Sentinel, Entra audit logs, Purview activity tracking.
Gaps
  • No per-session intent binding
  • Scope is service-level, not action-level
  • No behavioral authorization model
  • Copilot Studio connector credentials often long-lived in practice
  • No NHI offboarding workflow for decommissioned agents
AWS Bedrock Agents
Amazon
Identity Model
IAM execution roles assigned to each agent. Knowledge base access via S3 bucket policies. Lambda action groups carry their own IAM roles.
Credential Type
STS-issued temporary credentials via AssumeRole. Short-lived by default.
Scope Binding
IAM policies with resource-level conditions. No native concept of agent session scope. The role governs all actions regardless of session context.
Delegation
Inline agents (supervisor pattern) invoke sub-agents. Each sub-agent uses its own execution role; the supervisor role needs a trust policy to assume sub-agent roles.
Audit
CloudTrail (all API calls), Bedrock model invocation logging, optional S3 access logging.
Gaps
  • No session-level scope restriction
  • IAM policies are static; no runtime intent classification
  • Multi-agent role chains are difficult to audit end-to-end
  • No behavioral drift detection
  • Action group Lambda functions are separately governed and easily misconfigured
Google Vertex AI Agents
Google
Identity Model
GCP service accounts. Agent Builder (Dialogflow CX) uses per-agent service accounts with Workload Identity Federation available.
Credential Type
Service account keys or Workload Identity Federation tokens. WIF tokens are short-lived and keyless.
Scope Binding
GCP IAM roles and conditions. Resource-level control. VPC Service Controls add a network-level boundary.
Delegation
Tool use within a single agent session. Multi-agent via Reasoning Engine (Vertex AI): each sub-call uses the orchestrator's service account unless explicitly configured otherwise.
Audit
Cloud Audit Logs (Data Access, System Event, Policy Denied), Cloud Logging, BigQuery export.
Gaps
  • No native multi-agent identity segmentation
  • Single orchestrator service account often used for all sub-calls
  • No session intent binding
  • No behavioral authorization layer
  • WIF setup is non-trivial; many teams default to service account keys
Microsoft Copilot Studio
Microsoft
Identity Model
Connector-based. Each connector (SharePoint, Salesforce, etc.) has its own credential store. Bot Framework service principal handles channel authentication.
Credential Type
OAuth 2.0 tokens per connector, often with refresh tokens stored in the platform. Bot channel uses Azure Bot Service registration.
Scope Binding
Defined at connector configuration time. No runtime scope adjustment.
Delegation
Via Topics and Power Automate flows. Power Automate flows run under the connector user's credentials or a service account.
Audit
Power Platform admin center, Azure AD sign-in logs per connector.
Gaps
  • Heavy reliance on long-lived OAuth refresh tokens
  • No unified NHI inventory across connectors
  • Power Automate service accounts often carry broad permissions
  • No behavioral authorization model
03 / Open Standards 300

What open standards contribute

The standards below are the best currently available. Some are production-ready. Some are research proposals. All of them cover pieces of the problem. None cover the full stack.

SPIFFE / SPIRE CNCF Graduated
What it is

Secure Production Identity Framework for Everyone. X.509-based workload identity. SPIRE is the reference implementation.

Mechanism

Node and workload attestation. SVID issuance (X.509 or JWT). Short-lived certificates (default 1hr). Automatic rotation. Trust bundle federation across clusters.

Agent Identity Use

Issue a unique SVID per agent workload. Workload attestation ties identity to the deployment, not just the process. SVIDs expire automatically, no manual rotation needed.

Covers identity issuance and workload attestation. Does not address behavioral authorization or session-level scope binding.
A2A Protocol (Google) Open Spec / Early
What it is

Agent-to-Agent protocol for multi-agent delegation. Defines Agent Cards (capability declarations) and delegation flows.

Mechanism

Agent Cards describe what an agent can do. Delegation passes a task plus context to a sub-agent with declared scope. Sub-agent responds with result.

Agent Identity Use

Delegation chain traceability. The delegating agent declares the scope granted to the sub-agent.

Delegation is declarative, not enforced at runtime. No published eval suite for scope boundary enforcement. Fast-moving spec, no stable release.
Mission-Bound OAuth (McGuinness) Research Proposal
What it is

Extension to OAuth 2.0 for agentic systems. Credentials bound to a declared mission at issuance time.

Mechanism

Access tokens include a mission claim specifying the authorized purpose. Token validation checks declared mission against the requested action at the resource server.

Agent Identity Use

An agent presents a token with mission="process-expense-reports". Any action outside that scope is rejectable at the resource server.

Requires resource servers to validate mission claims. No widely adopted implementation. No IETF standardization process underway.
OWASP NHI Top 10 OWASP 2025
What it is

Top 10 security risks specific to Non-Human Identities. Directly applicable to agent identity management.

Key Risks

NHI1 (improper offboarding), NHI2 (secret sprawl), NHI3 (overprivileged NHI), NHI7 (long-lived secrets), NHI9 (NHI reuse), NHI10 (human use of NHI). NHI3 and NHI7 are the most common agent identity failures seen in production.

Descriptive, not prescriptive. Identifies risks without mandating controls or architectures. Useful for gap analysis, not implementation.
OWASP Agentic AI Top 10 OWASP 2026
What it is

Top 10 risks specific to agentic AI systems.

Key Risks

ASI01 (agent objective hijacking), ASI02 (tool misuse), ASI04 (excessive autonomy), ASI07 (multi-agent trust collapse), ASI08 (unauthorized data access), ASI09 (irreversible action), ASI10 (insufficient audit trail). ASI07 maps directly to the delegation chain gap.

Descriptive risk taxonomy. No prescriptive authorization architecture. Pairs well with OWASP NHI Top 10 for a full risk picture.
Singapore MGF (IMDA, 2026) Framework
What it is

Model Governance Framework, January 2026 edition. The most forward-looking agentic governance treatment in any major published framework to date.

Four Pillars

Principal Hierarchy, Scoped Permissions, Minimal Footprint, Auditable Decisions. Principal Hierarchy directly addresses multi-agent authorization chains. Scoped Permissions requires session-level scope binding, not service-level.

Framework-level guidance only. No implementation specification. Vendors have not adopted it as of early 2026.

05 / Reference Architectures 400

Three patterns worth implementing

These are not vendor-specific. They apply the controls from the sections above into implementable patterns. Start with Architecture 1, add Architecture 2 when you introduce multi-agent orchestration, add Architecture 3 for any irreversible or high-impact action.

Architecture 1: Single Agent with SPIFFE + Scoped Session Token

The SVID establishes workload identity. The session scope token restricts what the workload can do within this specific session. Both are short-lived. The combination enforces least privilege at the session level, not just the service level.

Architecture 2: Multi-Agent Delegation Chain with Scoped Sub-Tokens

Each hop in the chain carries only the scope it was granted. The sub-agent cannot exceed the orchestrator's scope, and the resource server validates both the SVID and the delegation token. The audit record captures the full chain, not just the leaf action.

Architecture 3: Human-in-the-Loop Approval Gate for Irreversible Actions

The approval gate fires before the irreversible step. Timeout defaults to abort, not execute. The execution token is scoped to the specific approved action, not the full session. The audit trail captures the full approval chain: who proposed, who approved, what was authorized, what executed.

06 / The Unaddressed Gap 400

Every platform above covers credential management. None cover behavioral authorization.

That is the gap. Not a missing feature, not a roadmap item. An architectural category that does not exist in any commercial platform or finalized standard today.

1

Static credentials cannot enforce dynamic intent

SPIFFE issues the identity. IAM binds the scope. Neither evaluates whether the agent's action in this moment, in this context, matches the declared purpose of the session. A customer service agent with read-and-write credentials is authorized to write. But not to write to this record, in response to this prompt, at this step in this workflow. That distinction does not exist in any current system.

2

Behavioral authorization is per-request, not per-issuance

DIRA (Dual-Intent Runtime Authorization) evaluates three signals per action: (a) what the agent declared it would do at session start, (b) what action it is attempting to take right now, (c) whether the attempted action falls within the authorized scope for this specific request. Three signals, one enforcement decision. Applied at the tool call, not at credential issuance. This is a different class of control from anything in the credential layer.

3

Audit without enforcement is forensics, not security

Every platform above logs. None block. There is a material difference between knowing that an agent performed an unauthorized action after the fact, and preventing the action before it executes. DIRA's enforcement mode pauses or denies the action at the point of behavioral divergence. Before the irreversible step. Not after the incident.

This is ongoing research. The behavioral authorization problem is solvable, and the architecture for solving it is the subject of active work at cyberdaemon.ai. Follow the thread there.
